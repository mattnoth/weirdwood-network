## What [LINK] Shapes Exist

The `[LINK]` token is generated by `scripts/wiki-pass2-build-cross-refs.py:94` in its
`extract_snippet()` function. For every wikilink `[anchor_text](wiki:Page)` in a prose
line, the builder outputs a snippet of context around the link — but replaces the link
itself with the literal string `[LINK]`. The actual display text of the link goes into a
separate `anchor_text` field.

**Every candidate row has exactly one `[LINK]` token** (it marks the row's own target).
Other links in the same snippet remain in `[text](wiki:Page)` format.

Examples from the completed batches:

```
snippet: "Ser [LINK] captured Jaime at the Whispering Wood."
anchor_text: "Brynden Tully"

snippet: "Dorden stays with Queen [LINK] at [Eastwatch-by-the-Sea](wiki:Eastwatch-by-the-Sea)..."
anchor_text: "Selyse Florent"

snippet: "...fought for [LINK] at the [Battle of the Blackwater](wiki:Battle_of_the_Blackwater)..."
anchor_text: "Stannis Baratheon"
```

**Counts in the 21 completed Sonnet batches:**
- 351 `emit_edge` rows with `[LINK]` in `evidence_snippet`
- 0 with `anchor_text` field (the classifier didn't preserve it — it copied `snippet` verbatim)

**Counts in queued candidate files:**
- 4,744 candidate files across 265 queued buckets
- 122,442 total candidate rows
- 122,404 rows with `[LINK]` in snippet (99.97% — effectively all rows)
- 38 rows without `[LINK]` (snippet was `null` or already plain text)

## Architecture Decision: Post-Processing Script

Three options were considered:

**Option A: Modify `build-cross-refs.py`** — Emit resolved text from the raw cross-refs stage. Rejected: `build-cross-refs.py` generates 91k+ rows for all wiki prose, and it emits `[LINK]` intentionally to mark the cross-reference position. Changing it would force a full re-crawl of all prose pages to regenerate `cross-references.jsonl`.

**Option B: Inline in `build-edge-candidates.py`** — Substitute during candidate generation. Selected for future runs. Cost: ~10-line addition. Limitation alone: doesn't fix the 4,744 already-generated queued candidate files.

**Option C: Standalone post-processing script** — Read existing candidate files, substitute in-place, write back atomically. Selected as the primary deliverable. Runs in seconds, idempotent, bucket-scoped for targeted reruns.

**Final answer: both B and C together.** The post-processor handles the 4,744 already-generated files. The inline edit in `build-edge-candidates.py` handles any future regeneration. Neither touches completed batches.

---

## Substitution Marker Choice

`[LINK]` → `«anchor_text»` (Unicode angle quotation marks, U+00AB and U+00BB).

Plain brackets `[...]` were rejected because they appear extensively in the same snippets for other wikilinks (`[Stannis Baratheon](wiki:Stannis_Baratheon)`) — the classifier would see two bracket-delimited tokens of different semantic types in the same snippet and might confuse them.

Angle quotes `«»` are visually distinct, safe in UTF-8 JSON, never appear in wiki prose, and clearly delimit a resolved name. The `anchor_text` field is preserved unchanged alongside the substituted snippet.

## Script Contract: `scripts/stage4-resolve-link-placeholders.py`

**Input:** Queued candidate JSONL files at:
```
working/wiki/pass2-buckets/<bucket>/prose-edge-candidates/<slug>.candidates.jsonl
```

**Scope filter:** Only processes buckets with `prose-edge-candidates/` but no completed
`prose-edges/*.edges.jsonl`. Completed batches (the 21 Sonnet control arm) are skipped.

**Transformation per row:**
```
snippet: "Ser [LINK] arrived at Riverrun."
anchor_text: "Brynden Tully"
→
snippet: "Ser «Brynden Tully» arrived at Riverrun."
anchor_text: "Brynden Tully"   (preserved)
```

**Modes:**
- `--plan` (default): dry-run, prints stats without writing
- `--apply`: rewrites files atomically (`.tmp` → rename)
- `--bucket BUCKET_ID`: restrict to one bucket

**Idempotency:** After `--apply`, re-running `--plan` shows 0 rows would be substituted.

**How to run:**
```bash
# Plan (check what would happen)
python3 scripts/stage4-resolve-link-placeholders.py

# Apply to all queued candidate files
python3 scripts/stage4-resolve-link-placeholders.py --apply

# Apply to one bucket only (smoke test or targeted regen)
python3 scripts/stage4-resolve-link-placeholders.py --apply --bucket characters-house-stark
```

**When to re-run:** If `wiki-pass2-build-edge-candidates.py --apply` is run again for new
buckets, run the substitution script afterward for any new queued buckets. Or rely on the
inline substitution now baked into the candidate generator (which does the substitution at
write time for any future run).

After:

  candidate dirs) — those are frozen; the classifier already consumed them and their outputs
