## Phase 2 — Recovery and Path 2 decision (post-crash)

The post-crash Claude (this session) ran the context-recovery checks from the continue prompt and found a surprise: the 6 direwolf nodes already on disk (timestamps 20:01-20:03) had the **disambiguated names already** — `nymeria-direwolf.node.md`, with `wiki_source: https://awoiaf.westeros.org/index.php/Nymeria_(direwolf)`. These weren't from a re-emit. They were from Session 20's original smoke run, where the agent self-corrected the buggy bundle.

The bucket_input.json (regenerated by triage --accept) correctly pointed at `Nymeria (direwolf)`. The validator-report.json showed `"passed": true`. The manifest was the only thing left in `version-stale` — because the launcher's status flip happens after a successful run, and no run had been attempted post-fix.

This reframed the remaining work. The continue prompt assumed a re-emit was necessary. But the on-disk state already reflected the correct content. Three paths:

1. **Re-emit anyway** — burn $1.15 / 5 min to validate the script-side fix end-to-end. Produces identical content (or close, with agent-prose variance).
2. **Skip re-emit, accept current nodes** — flip manifest to `complete`, resolve the question, worklog, archive.
3. **Re-emit only Nymeria** — not viable. The launcher promotes buckets atomically; per-page mode is real engineering.

Matt picked Path 2. Reasoning was implicit: the nodes are content-correct, the script fix can be exercised by the next bucket that has a similar shape, and burning $1.15 for a regression-test on a single bucket is wasteful when the next 36-bucket run will exercise the path naturally.

## Phase 3 — P2 fix surfaced

While confirming the manifest had flipped to `complete`, Matt selected line 28 of the manifest (`"validation_report": null`) and asked why. Quick investigation:

- The runbook (§5.2) schema specifies `"validation_report": "path/to/validator-output.json"`.
- `validator-report.json` exists on disk in every bucket dir post-run.
- `cmd_run` in `wiki-pass2.sh` runs the validator (writes the report) but **never updates the manifest's `validation_report` field**.
- Triage preserves whatever value is there (`existing.get("validation_report")`); reset nullifies it.
- Net: every existing manifest in `working/wiki-pass2/` had `validation_report: null`.

Matt also asked about `expected_edges`: confirmed not in v1 schema. Validator only checks `node_count_actual == node_count_expected`. Edges live in node bodies. `expected_edges` would be a v2 schema move.

The P2 fix took two lines in `cmd_run`: write the path on both the validation-pass branch and the validation-failed branch (the failed report is still useful forensically). Direwolves manifest backfilled by hand. All future bucket runs will populate this field automatically; the 508 not-yet-run manifests will heal on first run.

The trade-off Matt called out — "if we keep going it's just going to be a useless parameter" — was the deciding factor. Cheap to fix now, mass-rewrite later.

## What surprised

The crash was surprising. The Claude that wrote the continue prompt assumed the re-emit hadn't run; the Claude that read the continue prompt found Session 20's self-corrected output sitting on disk and had to re-derive what the actual remaining work was. The continue prompt was correct about the code edits, manifest state, and unresolved question — but the assumption that we needed to re-emit was wrong because Session 20's agent had already done the work. The lesson: when reading a continue prompt after a crash, run the recovery checks before executing the plan. The plan in a crash-recovery prompt is a draft, not a contract.

The P2 fix surfaced organically — Matt highlighting line 28 of the manifest. This is the value of curating output: a quick visual scan caught a schema-vs-implementation gap that the validator never would have flagged. Worth keeping these "what does this null mean" reviews as a routine pre-launch step.
